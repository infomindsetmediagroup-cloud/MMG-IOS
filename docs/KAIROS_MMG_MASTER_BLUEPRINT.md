## 4. Three-Layer Kairos Intelligence Architecture

Kairos uses a three-layer intelligence model.

### 4.1 Public Knowledge Brain

The Public Knowledge Brain contains approved public MMG knowledge, including:

- Products
- Services
- Knowledge Library content
- FAQs
- Policies
- Public documentation
- Kairos Moments
- Navigation content
- Public business information

Website assistant responses should be grounded in this approved knowledge base.

### 4.2 Customer Intelligence

For authenticated users, Kairos maintains customer-specific continuity using that customer’s own data, subject to privacy and access controls.

This may include:

- Purchases
- Vault/library access
- Prior conversations
- Projects
- Preferences
- Activity
- Subscription profile
- Recommended content

Customer information must never leak between customers.

### 4.3 Founder Intelligence

Founder Intelligence is private owner-level memory and operational intelligence for MMG/Kairos development. It includes strategy, backlog, operating doctrine, implementation decisions, and business planning. This layer is not public and is not exposed to customers.

## 5. Progressive Autonomy Architecture

Kairos should be developed as a progressive autonomy platform.

### Phase 1 — Launch

AI-guided operations without requiring a dedicated always-on server from day one.

Kairos helps manage:

- Planning
- Prioritization
- Product generation
- Documentation
- Pricing
- Knowledge Bank organization

- Shopify preparation
- Operational workflows

### Phase 2 — Hybrid

Introduce backend services for:

- Scheduled jobs
- Integrations
- Synchronization
- Customer vault functionality

### Phase 3 — Enterprise

Add continuous autonomous capabilities through dedicated backend infrastructure, including:

- Monitoring
- Event-driven workflows
- Reporting
- Advanced integrations

All systems should be designed so later phases extend the architecture rather than forcing a rebuild.

- Place a natural-language command block near the top of the Kairos dashboard.

## 10. Customer Portal and Subscription Intelligence

Personalized content extends into the customer dashboard, not only onboarding.

During account creation or subscription enrollment, customers complete a lightweight profile covering:

- Profession
- Use case
- Interests
- Preferred content categories

Kairos should continuously curate recommended content based on this profile.

Before scheduled subscription distribution, subscribers should be able to review the proposed content package and swap eligible titles within their interests.

The target selection window is 24–48 hours before distribution. If no changes are made before cutoff, Kairos sends the default recommended package.

## 11. Publishing Architecture

Kairos/MMG maintains two separate canonical Shopify publishing workflows:

1. Digital Download Products
2. Service Products

### Digital Download Products

MMG products are digital downloads and should not be treated as ordinary physical listings.

Checklist:

- Create or update Shopify product
- Configure as digital-download product
- Attach full digital package through the Shopify digital-download workflow
- Include primary PDF/product file
- Include cover image and supporting assets
- Validate delivery workflow
- Confirm product metadata and SEO fields

### Service Products

Service products may be created manually for quality control because they include multiple variants and require careful setup.

Service listings must include the canonical onboarding PDF. The user may upload/provide that onboarding PDF so ChatGPT/Kairos can reuse it.

## 15. GitHub Workflow Standard

Avoid burning GitHub Actions minutes during active development whenever possible.

Standards:

- Batch changes into meaningful commits
- Avoid repeated deploy-triggering pushes
- Prefer non-deploy branches for active work
- Use workflow-skip mechanisms when appropriate
- Preserve GitHub Actions minutes for final validation/deployment
- Use manual deployment triggers where practical
- Do not expose secrets

The canonical GitHub Actions secret name is `KAIROS`.

Workflow authentication should reference:

```yaml
GITHUB_TOKEN: ${{ secrets.KAIROS }}
```

Checkout should use:

```yaml
with:
  token: ${{ secrets.KAIROS }}
```

Only validate that the secret exists; never print, echo, store, retrieve, or expose the raw token value.
